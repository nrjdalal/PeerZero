PeerZero runs three processes locally, started together by `bun run dev`:

| Service    | Runtime           | Role                                               |
| ---------- | ----------------- | -------------------------------------------------- |
| **Web**    | Next.js (Bun)     | The web UI you interact with.                      |
| **API**    | Hono (Bun)        | A thin seam in front of the download engine.       |
| **Engine** | Node + WebTorrent | The actual BitTorrent client that downloads files. |

## Why a separate download engine?

The download engine is a small **Node process**, not part of the Bun API. WebTorrent
depends on a native addon (`node-datachannel`) that crashes under Bun, so it runs in
its own Node sidecar and exposes a tiny local HTTP API that the Hono backend proxies
to. WebTorrent is also pinned to a version known to complete downloads cleanly.

Keeping the engine behind one seam (`api/hono/src/lib/torrent/engine.ts`) means the
rest of the app doesn't care how downloading works - a different backend could be
swapped in without touching the UI.

## How data flows

**Downloading:**

```
Add magnet → Hono → engine (WebTorrent) → peers/DHT/trackers → files on disk
```

**Live progress:** the engine emits a full snapshot roughly once per second. The web
app holds **one WebSocket** for the whole session, so navigating around the app never
drops or reconnects the feed.

## State that persists

- **Torrents** - the engine remembers what you've added (and each torrent's
  metadata), so they're restored when it restarts.
- **UI preferences** - sort order and visible columns are saved in the browser (via a
  small persisted store) and restored on reload.

## Tuning

The engine reads a few environment variables - most usefully `TORRENT_MAX_CONNS`
(per-torrent connection limit, default `25`, kept low on purpose). Too many
connections overwhelm a weak router and degrade the whole network. See
[Troubleshooting](/docs/troubleshooting).
